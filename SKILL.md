# 微信公众号文章写作

把素材变成适合手机阅读、值得收藏和转发的完整公众号草稿。先抓住读者，再交付价值，最后只推动一个明确动作。支持一键发布到公众号草稿箱。

## 默认目标

除非用户明确只要提纲、标题或修改建议，否则直接交付成稿，不停在分析和写作建议。

优先根据上下文推断缺失信息，并在输出开头说明假设。只有缺失信息会明显改变文章方向、造成事实错误或引发合规风险时才追问。

### 启动时先读配置

Skill 触发时，先检查是否存在用户配置文件，读取后覆盖默认参数。配置文件优先级：

1. 当前项目根目录 `.wechat-writer-config.json`
2. 用户主目录 `~/.config/wechat-writer/config.json`
3. 内置默认值

详细配置项读取 `references/user-config.md`。环境变量 `WECHAT_APP_SECRET` 优先级高于配置文件。

默认参数（可被配置文件覆盖）：

- 文章类型：根据素材判断，默认技术科普或项目分享
- 目标读者：泛技术爱好者
- 核心目标：涨粉 + 互动
- 作者调性：独立开发者（`indie-dev`）
- 输出形态：可直接粘贴到公众号编辑器的完整草稿
- 配图模式：auto（有图片工具则生成，否则纯文字降级）

## 工作流

### 1. 读取并核实素材

优先读取用户提供的文档、网页、代码、截图、提交记录、测试结果、运行日志或项目文件。涉及技术项目时，至少确认核心实现、真实结果、一次实际操作和一个真实坑点，不要凭主题脑补。

引用数据、政策、平台规则或时效性事实时先核实来源。无法核实时删掉具体数字，或在发布前检查中明确标记待确认项，绝不编造来源。

如果用户只提供主题、提纲或少量参考文档，不要把素材不足直接交还用户。先主动补全：

1. 列出文章还缺的事实、数据、案例、实战过程和坑点
2. 使用可用的检索工具查找权威来源、真实案例、公开复盘和最新数据
3. 交叉核对关键结论，记录来源主体、发布日期、统计口径和适用范围
4. 把外部案例写成“某机构/某项目的实践”，不能改写成作者亲历
5. 只有检索后仍缺少决定文章方向的关键信息时，才向用户追问

**真实实战是完整成稿的硬门槛。** 每篇完整文章至少包含：

- 1 段有来源的实战经验：具体场景、实际动作、关键判断和可观察结果
- 1 个真实坑点：触发场景、故障现象、原因、处理办法和修复后的验证结果

真实实践可以来自作者或用户的亲历，也可以来自有明确出处的外部案例。如果现有素材不足，先继续读取项目资料并主动检索补全；仍然不足时，再向用户追问最少量的信息。不能把行业常识、推测或虚构故事改写成“我做过”“我踩过”。详细标准读取 `references/writing-playbook.md` 的“资料研究与数据补全”和“真实实战与坑点”章节。

#### 时间与经验真实性（硬性约束）

**禁止编造使用时长或经验深度。** 以下写法属于虚构，一律禁止：

- "用了半年""深度使用三个月""上线一年以来"——除非用户明确提供了这个信息
- "根据我长期使用的经验"——如果没有素材支撑，这是凭空捏造
- 用虚构时长建立"资深感"，是最常见、也最容易被识破的 AI 腔

**写新产品或新工具的文章前，必须先核实其发布时间。** 如果产品刚发布不久，作者不可能"用了很久"，强行写出来会直接暴露内容是编造的。

核实方法：
1. 检索产品官网、官方公告或权威媒体报道，确认首次公开发布日期
2. 用当前日期减去发布日期，得出产品存在时长
3. 文中经验描述不得超过产品存在时长，也不得超过用户实际提供的信息

**资深视角靠分析深度，不靠使用时长。** 正确的做法：

- 通过"用错了才知道的判断"体现深度：描述一个反常识的决策点或坑点
- 通过"为什么这样选，放弃了什么"体现取舍意识
- 通过具体的操作细节（配置项名称、命令、报错信息）体现真实性
- 通过"它在哪里不好用"体现独立判断，而不是只说优点

时长可以模糊处理：用"深度用下来""认真配置过""实际跑过"等不带具体时间的表达，比编造"半年"更诚实，也更耐读。

### 2. 确定写作参数

明确四项：

1. 文章类型：技术科普 / 教程干货 / 项目复盘 / 行业观察 / 产品推广
2. 目标读者：泛大众 / 泛技术爱好者 / 专业开发者 / 行业从业者
3. 核心目标：涨粉 / 互动 / 转化 / 品牌曝光 / 知识沉淀
4. 作者调性：独立开发者 / 硬核技术 / 推广种草

需要选择结构、标题、语气和 CTA 时，读取 `references/writing-playbook.md`。不要把整个 reference 复述给用户，只执行与当前文章相关的规则。

### 3. 设计传播抓手

每篇至少落实以下四项：

- **粘性**：前三屏持续出现新信息、真实细节或自然问题
- **流通**：提供一个值得收藏的清单、模板、对比或结论卡片
- **扩散**：让读者能用一句话解释为什么转发给别人
- **易读**：只看小标题和加粗句，也能理解约 70% 的主线

不要为了传播制造虚假悬念。标题承诺必须在正文兑现。

### 4. 先搭正文骨架

正文必须包含：

- 1-3 段的痛点、场景、结论或具体故事开头
- 2-5 个顺序明确的主体模块
- 至少一段真实实战经验，包含场景、动作、判断和结果
- 至少一个真实坑点，讲清现象、原因、处理和验证
- 用代码、日志、截图、提交、测试结果或用户明确陈述支撑实战内容
- **CTA 前必须有 1-2 句思考收束**：先回扣主题核心判断或向外延伸一个开放问题，再接 CTA，不允许正文直接跳进互动问句
- 一个与核心目标一致的 CTA，只推动一个动作

禁止宏大背景、文章目录预告、重复全文的总结和空洞的“欢迎关注”。

### 5. 决定配图模式

> ⚠️ **配图决策必须在完整正文草稿完成后执行。禁止在正文写完之前规划 Shot List 或生成任何图片。** 正文写完后，逐段过配图价值自检，由文章内容决定图片数量，不预设数量。

先判断当前环境是否有可用的图片生成或编辑工具，再选择模式：

#### 完整图文模式

满足以下任一条件时使用：

- 用户明确要求配图、封面或“可直接发布”图文稿
- 当前环境有图片工具，且用户没有要求只输出文字

执行规则：

1. 对照 `references/design-system.md` 判断哪些内容值得可视化
2. 叙事、比喻、情绪场景读取 `references/cartoon-illustrations.md`
3. 流程、架构、原理、对比读取 `references/flowchart-style.md`
4. **封面图是完整图文模式的必须输出**：读取 `references/cover-image.md`, 无论用户是否额外要求，都必须生成封面图并在成稿中标注
5. 优先使用用户提供的真实截图和素材
6. 实际生成所有图片，并用 `![说明](路径)` 嵌入正文和封面对应位置

如果用户尚未选标题，默认使用标记为“推荐”的标题生成封面，不中断交付等待选择。

#### 纯文字降级模式

图片工具不可用，或用户明确只要文字稿时使用。用以下格式提供可执行的配图说明：

```markdown
> 【配图建议：图片类型；画面内容；关键标注；建议比例】
```

在发布前检查中说明“图片尚未生成”。不要声称占位图已经生成。

### 6. 完稿并自检

> ⚠️ **无论是新建文章还是经过多轮迭代修改，只要输出完整正文，就必须完整跑完以下检查。不允许因为"只是在修改"而跳过排版步骤。坑点结构必须使用 `wechat-layout.md` 规定的信息标签格式（触发场景 / 看到的现象 / 根本原因 / 实际处理 / 怎么避免），不允许把坑点嵌在普通段落里。**

生成完整文章时读取 `references/wechat-layout.md`，按公众号编辑器和手机阅读规则完成语义排版，再执行以下检查：

完成正文后检查：

- 前 3 行没有报幕、铺垫或宏大叙事
- 小标题连起来能看出主线
- 段落适合手机扫读，没有连续文字墙
- 表格、长代码和多层列表已转换为手机友好的列表、短代码或图片
- 图片紧跟相关内容并有说明，复制到编辑器后仍能看懂正文结构
- 关键结论有事实、案例或来源支撑
- 实战经验来自可识别的项目素材、用户陈述或注明出处的外部案例，不是通用常识换成第一人称
- 外部数据注明时间、范围、口径和来源；不同口径的数据不强行比较
- 坑点讲清“什么场景出问题、看到了什么、为什么、怎么修、如何确认修好”
- 专业术语先用大白话解释，读者不看代码也能理解问题和解决过程
- 没有“此外”“综上所述”“革命性的”等 AI 腔和宣传腔
- 中英文、中文与数字之间留空格，专有名词写法正确
- 标题、摘要、正文、图片和 CTA 承诺一致
- 没有极限词、虚假承诺、无出处数据和诱导分享
- 粘性、流通、扩散、易读四项都有具体抓手
- **没有编造使用时长**：文中出现"用了 N 个月/年"类表达，必须能对应用户提供的真实信息或可核实的产品发布时间；否则删除或改为不带时间的模糊表达
- **产品发布时间已核实**：涉及新产品时，已确认其发布时间，文中经验描述未超出合理范围

### 7. 发布到公众号草稿箱（可选）

用户明确要求"发布""发到公众号""存草稿箱"时，执行本步骤。否则只输出文章，不主动发布。

执行前先确认：

1. 当前环境是否有 `tools/publish_to_wechat.py` 脚本
2. 凭证是否已配置（环境变量 `WECHAT_APP_ID` / `WECHAT_APP_SECRET`，或配置文件中的 wechat 节）
3. 封面图是否已生成或用户提供

缺少凭证时，列出需要用户补齐的内容，不要继续尝试。

文章 Markdown 首行必须是 `# 标题`；如果文件中找不到一级标题，停止发布并提示用户补充标题行，不尝试用空标题提交。

发布流程：

1. 将完稿的 Markdown 保存为临时文件
2. 如果有封面图，保存为临时图片文件
3. 执行发布脚本：`python tools/publish_to_wechat.py <文章> --cover <封面>`
4. 成功：向用户返回 media_id 和草稿箱查看指引
5. 失败（40164）：告知用户 IP 不在白名单，给出需要添加的 IP 和操作路径，等待用户确认后重试
6. 失败（其他错误）：展示原始错误，给出排查建议

详细发布规则和错误处理读取 `references/wechat-publish.md`。

## 修改模式

当用户对已输出的文章提出修改要求时，进入修改模式，不重新走完整写作流程。

修改模式触发关键词：改、修改、调整、缩短、拉长、换、再改、不对、不喜欢，以及所有针对已输出内容的局部反馈。

修改模式核心原则：
- **最小改动**：只改用户明确要求的部分，不顺手重写全文
- **一致性维护**：改了标题要检查摘要和 CTA，改了论点要检查前后衔接
- **完整自检**：只要输出完整正文，就必须跑完全部发布前检查
- **修改留痕**：在输出末尾（发布前检查**之后**）增加"本次修改说明"，1-3 条说清改了什么

详细修改工作流、常见场景处理和修改轮次边界读取 `references/iteration-mode.md`。

## 输出格式

每次生成完整文章时严格按以下顺序输出：

1. **使用假设**：一行说明文章类型、目标读者、核心目标和作者调性
2. **备选标题**：5 个（或按配置文件 `output.title_count`），覆盖不同角度；选出 1 个并标记"推荐"
3. **摘要**：40-80 字（或按配置文件区间），可直接用作公众号摘要
4. **正文**：完整可发布文本，包含小标题、重点、图片或配图建议和结尾 CTA；使用外部资料时，在正文末尾增加"资料来源"
5. **转发文案**：1-2 条（或按配置文件 `output.forward_copy_count`）适合朋友圈或社群的自然短句
6. **发布前检查**：只列确实需要用户补齐或确认的链接、来源、敏感表达和未生成图片；没有则写"无"
7. **本次修改说明**（仅修改模式）：1-3 条短句说明本次改动

不要把内部评分、思维过程、写作框架说明或 reference 内容放进成稿。

## 文章类型路由

| 类型 | 核心结构 | 必须交付 |
|------|----------|----------|
| 技术科普 | 痛点 → 大白话解释 → 原理 → 实战 → 坑点 → 边界 | 真实使用场景、坑点、原理图或配图建议 |
| 教程干货 | 目标 → 前置条件 → 步骤 → 实测 → 踩坑 → 清单 | 每步的动作与结果、真实坑点、可收藏清单 |
| 项目复盘 | 结果 → 动机 → 决策 → 问题 → 修复 → 验证 → 下一步 | 真实结果、工程取舍、完整坑点链路 |
| 行业观察 | 观点 → 实战案例 → 坑点 → 证据 → 边界 → 对读者的意义 | 可核实案例、实践教训、明确立场和成立边界 |
| 产品推广 | 真实问题 → 实际使用 → 坑点 → 产品 → 证据 → 边界 | 真实试用过程、问题与处理、场景收益和克制 CTA |

详细结构模板和写法读取 `references/writing-playbook.md`。

## 图片规则摘要

- 图片只在能提高理解、记忆或可信度时生成，不做纯装饰
- 微信正文不使用 Mermaid、轮播、标签页或折叠组件
- 微信正文默认不使用 Markdown 表格、复杂 HTML/CSS、多栏布局和自定义字体
- 多张相关截图优先竖向拼成长图，避免横向压缩
- 橘猫是每张正文图的**必须元素**：生成任何正文图片前，必须先为橘猫找到一个功能性角色——流程发起者、结果接收者、状态对比体验者、概念使用者等
- 加猫前必须写出角色说明："它代表谁、正在做什么、帮助理解哪个概念或流程"；写出来后才能生成图片。如果反复找不到合理角色，说明图的设计本身需要调整，而不是跳过橘猫
- 禁止把橘猫当背景、角落贴纸、品牌水印或与主题无关的可爱装饰
- 叙事插图由橘猫承担核心动作；流程图和原理图默认在流程起点或终点加猫；每类图都必须有橘猫，只有节点极度密集导致橘猫无处站立时才例外
- **图片 prompt 禁止输出任何系统内部标识文字**：`[GLOBAL STYLE — WeChat-Insight-Skill]`、`GLOBAL STYLE`、`WeChat-Insight-Skill` 等字符串不能出现在发给图像模型的 prompt 里，否则会被渲染成图片内的水印文字。正确做法是把 `design-system.md` 中的视觉 token 内容**直接内联**写进 prompt，去掉方括号标签头。

## 使用边界

本技能适合面向"刷到了顺便看看"的微信读者。如果目标是给专业开发者提供可运行代码、完整架构和系列化深度内容，应优先写技术博客，再按需要提炼公众号版本。

发布到公众号草稿箱需要已认证的公众号、AppID/AppSecret 和 IP 白名单配置，缺一不可。详细前置条件读取 `references/wechat-publish.md`。
