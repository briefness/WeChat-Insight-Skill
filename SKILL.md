# 微信公众号文章写作

把素材变成适合手机阅读、值得收藏和转发的完整公众号草稿。先抓住读者，再交付价值，最后只推动一个明确动作。

## 默认目标

除非用户明确只要提纲、标题或修改建议，否则直接交付成稿，不停在分析和写作建议。

优先根据上下文推断缺失信息，并在输出开头说明假设。只有缺失信息会明显改变文章方向、造成事实错误或引发合规风险时才追问。

默认参数：

- 文章类型：根据素材判断，默认技术科普或项目分享
- 目标读者：泛技术爱好者
- 核心目标：涨粉 + 互动
- 作者调性：独立开发者
- 输出形态：可直接粘贴到公众号编辑器的完整草稿

## 工作流

### 1. 读取并核实素材

优先读取用户提供的文档、网页、代码、截图、提交记录、测试结果、运行日志或项目文件。涉及技术项目时，至少确认核心实现、真实结果、一次实际操作和一个真实坑点，不要凭主题脑补。

引用数据、政策、平台规则或时效性事实时先核实来源。无法核实时删掉具体数字，或在发布前检查中明确标记待确认项，绝不编造来源。

如果用户只提供主题、提纲或少量参考文档，不要把素材不足直接交还用户。先主动补全：

1. 列出文章还缺的事实、数据、案例、实战过程和坑点
2. 使用可用的检索工具查找权威来源、真实案例、公开复盘和最新数据
3. 交叉核对关键结论，记录来源主体、发布日期、统计口径和适用范围
4. 把外部案例写成“某机构/某项目的实践”，不能改写成作者亲历
5. 只有检索后仍缺少决定文章方向的关键信息时，才向用户追问

**真实实战是完整成稿的硬门槛。** 每篇完整文章至少包含：

- 1 段有来源的实战经验：具体场景、实际动作、关键判断和可观察结果
- 1 个真实坑点：触发场景、故障现象、原因、处理办法和修复后的验证结果

真实实践可以来自作者或用户的亲历，也可以来自有明确出处的外部案例。如果现有素材不足，先继续读取项目资料并主动检索补全；仍然不足时，再向用户追问最少量的信息。不能把行业常识、推测或虚构故事改写成“我做过”“我踩过”。详细标准读取 `references/writing-playbook.md` 的“资料研究与数据补全”和“真实实战与坑点”章节。

### 2. 确定写作参数

明确四项：

1. 文章类型：技术科普 / 教程干货 / 项目复盘 / 行业观察 / 产品推广
2. 目标读者：泛大众 / 泛技术爱好者 / 专业开发者 / 行业从业者
3. 核心目标：涨粉 / 互动 / 转化 / 品牌曝光 / 知识沉淀
4. 作者调性：独立开发者 / 硬核技术 / 推广种草

需要选择结构、标题、语气和 CTA 时，读取 `references/writing-playbook.md`。不要把整个 reference 复述给用户，只执行与当前文章相关的规则。

### 3. 设计传播抓手

每篇至少落实以下四项：

- **粘性**：前三屏持续出现新信息、真实细节或自然问题
- **流通**：提供一个值得收藏的清单、模板、对比或结论卡片
- **扩散**：让读者能用一句话解释为什么转发给别人
- **易读**：只看小标题和加粗句，也能理解约 70% 的主线

不要为了传播制造虚假悬念。标题承诺必须在正文兑现。

### 4. 先搭正文骨架

正文必须包含：

- 1-3 段的痛点、场景、结论或具体故事开头
- 2-5 个顺序明确的主体模块
- 至少一段真实实战经验，包含场景、动作、判断和结果
- 至少一个真实坑点，讲清现象、原因、处理和验证
- 用代码、日志、截图、提交、测试结果或用户明确陈述支撑实战内容
- 一个与核心目标一致的 CTA，只推动一个动作

禁止宏大背景、文章目录预告、重复全文的总结和空洞的“欢迎关注”。

### 5. 决定配图模式

先判断当前环境是否有可用的图片生成或编辑工具，再选择模式：

#### 完整图文模式

满足以下任一条件时使用：

- 用户明确要求配图、封面或“可直接发布”图文稿
- 当前环境有图片工具，且用户没有要求只输出文字

执行规则：

1. 对照 `references/design-system.md` 判断哪些内容值得可视化
2. 叙事、比喻、情绪场景读取 `references/cartoon-illustrations.md`
3. 流程、架构、原理、对比读取 `references/flowchart-style.md`
4. 封面读取 `references/cover-image.md`
5. 优先使用用户提供的真实截图和素材
6. 实际生成图片，并用 `![说明](路径)` 嵌入正文对应位置

如果用户尚未选标题，默认使用标记为“推荐”的标题生成封面，不中断交付等待选择。

#### 纯文字降级模式

图片工具不可用，或用户明确只要文字稿时使用。用以下格式提供可执行的配图说明：

```markdown
> 【配图建议：图片类型；画面内容；关键标注；建议比例】
```

在发布前检查中说明“图片尚未生成”。不要声称占位图已经生成。

### 6. 完稿并自检

生成完整文章时读取 `references/wechat-layout.md`，按公众号编辑器和手机阅读规则完成语义排版，再执行以下检查：

完成正文后检查：

- 前 3 行没有报幕、铺垫或宏大叙事
- 小标题连起来能看出主线
- 段落适合手机扫读，没有连续文字墙
- 表格、长代码和多层列表已转换为手机友好的列表、短代码或图片
- 图片紧跟相关内容并有说明，复制到编辑器后仍能看懂正文结构
- 关键结论有事实、案例或来源支撑
- 实战经验来自可识别的项目素材、用户陈述或注明出处的外部案例，不是通用常识换成第一人称
- 外部数据注明时间、范围、口径和来源；不同口径的数据不强行比较
- 坑点讲清“什么场景出问题、看到了什么、为什么、怎么修、如何确认修好”
- 专业术语先用大白话解释，读者不看代码也能理解问题和解决过程
- 没有“此外”“综上所述”“革命性的”等 AI 腔和宣传腔
- 中英文、中文与数字之间留空格，专有名词写法正确
- 标题、摘要、正文、图片和 CTA 承诺一致
- 没有极限词、虚假承诺、无出处数据和诱导分享
- 粘性、流通、扩散、易读四项都有具体抓手

## 输出格式

每次生成完整文章时严格按以下顺序输出：

1. **使用假设**：一行说明文章类型、目标读者、核心目标和作者调性
2. **备选标题**：5 个，覆盖不同角度；选出 1 个并标记“推荐”
3. **摘要**：40-80 字，可直接用作公众号摘要
4. **正文**：完整可发布文本，包含小标题、重点、图片或配图建议和结尾 CTA；使用外部资料时，在正文末尾增加“资料来源”
5. **转发文案**：1-2 条适合朋友圈或社群的自然短句
6. **发布前检查**：只列确实需要用户补齐或确认的链接、来源、敏感表达和未生成图片；没有则写“无”

不要把内部评分、思维过程、写作框架说明或 reference 内容放进成稿。

## 文章类型路由

| 类型 | 核心结构 | 必须交付 |
|------|----------|----------|
| 技术科普 | 痛点 → 大白话解释 → 原理 → 实战 → 坑点 → 边界 | 真实使用场景、坑点、原理图或配图建议 |
| 教程干货 | 目标 → 前置条件 → 步骤 → 实测 → 踩坑 → 清单 | 每步的动作与结果、真实坑点、可收藏清单 |
| 项目复盘 | 结果 → 动机 → 决策 → 问题 → 修复 → 验证 → 下一步 | 真实结果、工程取舍、完整坑点链路 |
| 行业观察 | 观点 → 实战案例 → 坑点 → 证据 → 边界 → 对读者的意义 | 可核实案例、实践教训、明确立场和成立边界 |
| 产品推广 | 真实问题 → 实际使用 → 坑点 → 产品 → 证据 → 边界 | 真实试用过程、问题与处理、场景收益和克制 CTA |

详细结构模板和写法读取 `references/writing-playbook.md`。

## 图片规则摘要

- 图片只在能提高理解、记忆或可信度时生成，不做纯装饰
- 微信正文不使用 Mermaid、轮播、标签页或折叠组件
- 微信正文默认不使用 Markdown 表格、复杂 HTML/CSS、多栏布局和自定义字体
- 多张相关截图优先竖向拼成长图，避免横向压缩
- 橘猫只能作为功能性角色出现：必须代表具体人物、状态或流程角色，并通过动作帮助解释主题
- 加猫前用一句话说清“它代表谁、正在做什么、对理解有什么帮助”；答不出来就不加
- 禁止把橘猫当背景、角落贴纸、品牌水印或与主题无关的可爱装饰
- 叙事插图通常由同一只橘猫承担核心动作；流程图和封面则根据内容判断，不强制出现

## 使用边界

本技能适合面向“刷到了顺便看看”的微信读者。如果目标是给专业开发者提供可运行代码、完整架构和系列化深度内容，应优先写技术博客，再按需要提炼公众号版本。
