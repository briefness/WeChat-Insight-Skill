## 内容规划

先确定模块、顺序和删减项，再写正文。

所有文章都需要：

| 模块 | 作用 | 约束 |
|------|------|------|
| 痛点或钩子 | 让读者决定继续读 | 1-3 段，不超过 100 字 |
| 核心答案 | 兑现标题承诺 | 尽量前置，不故意拖延 |
| 主体内容 | 交付可复用价值 | 每节只解决一个问题 |
| 结尾 CTA | 推动下一步动作 | 评论、收藏、转发、关注或转化只选一个 |

内容密度判断：

- 一个点能展开 200 字以上且有独立价值，单独成节
- 多个点都不足 100 字，合并为列表或同一小节
- 只是顺带提及且不影响主线，删除

### 落地内容硬标准

完整文章不是“把主题讲完”，而是让读者知道下一步怎么判断、怎么操作、怎么验证、怎么避坑。每个主体小节必须自然落到至少一个可执行单元，整篇文章必须形成当前文章类型对应的完整落地链路。

允许的可执行单元只有以下 8 类：

| 类型 | 合格标准 | 适合场景 |
|------|----------|----------|
| 操作步骤 | 读者能照着完成动作，并知道预期结果 | 教程、工具使用、发布流程 |
| 判断标准 | 读者能据此选择方案或判断是否继续 | 选型、行业观察、产品推广 |
| 检查清单 | 读者能逐项自检，不遗漏关键条件 | 发布前、上线前、迁移前 |
| 避坑链路 | 包含触发场景、现象、原因、处理、验证和避免方法 | 复盘、教程、技术科普 |
| 对比表 | 讲清方案、成本、适用场景和风险 | 选型、产品推广、方法比较 |
| 最小实验 | 用小样例、命令、测试条件或截图验证一个判断 | 技术科普、教程、复盘 |
| 可复用模板 | 给出提示词、配置片段、表单结构、脚本骨架或文章结构 | 教程、内容生产、工作流 |
| 读者迁移问题 | 帮读者把经验套到自己的场景，而不是泛泛互动 | 观察、复盘、结尾 CTA |

每篇完整文章至少出现 3 类不同的可执行单元。主体小节可以自然呈现，不必机械标注“操作步骤”“判断标准”。教程、清单、复盘类文章可以显式加标签；观点文、产品文、行业观察应把可执行单元融入段落，避免模板感。

### 完整落地链路

不同文章类型的“落地”形态不同，不要把所有文章都写成同一种操作教程。

| 类型 | 必须形成的链路 | 不合格表现 |
|------|----------------|------------|
| 技术科普 | 现象 → 人话解释 → 最小实验或例子 → 适用边界 → 读者自查方法 | 只解释概念，不告诉读者怎么判断自己是否遇到这个问题 |
| 教程干货 | 前置条件 → 操作步骤 → 每步结果 → 验证方法 → 坑点处理 → 检查清单 | 只列步骤，不说明每步成功后应该看到什么 |
| 项目复盘 | 目标约束 → 方案选择 → 实际实现 → 问题链路 → 修复验证 → 可复用经验 | 只讲做了什么，不讲为什么这样选、怎么确认有效 |
| 行业观察 | 明确观点 → 真实案例 → 证据 → 不成立边界 → 对读者的判断方法 | 只堆趋势词和宏观判断，没有案例和边界 |
| 产品推广 | 真实问题 → 旧方案代价 → 实际试用 → 具体变化 → 不适用场景 → 下一步动作 | 只写功能卖点，不写使用证据和不适用场景 |

主体模块默认控制在 3-4 个。确实有多个必要点时可以写长，但不能为了覆盖面保留浅小节；素材不足以支撑证据、案例、边界和可执行单元的模块，必须删除或合并。

### 核心观点五层展开

每个核心观点至少回答 5 个问题：

1. 结论是什么？
2. 为什么是这样？
3. 有什么真实证据、案例、日志、数据或操作结果支撑？
4. 这个结论在什么情况下不成立？
5. 读者怎么把它用到自己的场景？

答不出第 3 个问题时，不能写成确定结论。可以降级成“待验证判断”，也可以删除。答不出第 4 个问题时，容易变成过度承诺，必须补充边界或改弱表达。

### 段落功能检查

文章可以长，但每段必须承担至少一种功能：

- 给出结论
- 解释原因
- 补充证据
- 展示操作
- 说明判断标准
- 交代边界
- 承接上下文并带来新信息
- 引导读者完成下一步动作

如果一段删掉后不影响理解、不减少证据、不影响操作，就删。两段表达同一个意思时合并。只是“因此我们可以看到”“这也说明了”等转场句，删掉或并入前后段。

主动删除：

- “随着 AI 技术的发展”等宏大背景
- “本文将分为以下几部分”等目录预告
- 重复正文的总结段
- “感谢阅读，欢迎关注”等固定套话
- 超出实际风险所需的长篇免责声明

## 标题写法

标题核心信息放前面，承诺必须能被正文兑现。一次给 5 个角度不同的标题，并标记一个推荐项。

常见角度：

| 角度 | 写法 | 适用场景 |
|------|------|----------|
| 平实型 | 直接说明做了什么 | 独立开发者、项目复盘 |
| 利益型 | 说明读者能省什么或得到什么 | 教程、推广 |
| 悬念型 | 留下具体但不虚假的问题 | 故事、行业观察 |
| 收藏型 | 数字清单、模板、避坑指南 | 教程干货 |
| 转发型 | 点出某类人的共同处境 | 观点、行业观察 |

独立开发者标题优先平实陈述：

- `我用 Python 写了个自动化工具，把重复流程跑通了`
- `Seedance + FFmpeg：从脚本到成片，我是怎么做的`
- `周末项目：用一个 API Key 串起完整工作流`

避免：

- 正文无法兑现的标题党
- “震惊”“必看”“颠覆”等廉价刺激
- 泛读者看不懂的纯英文或生僻术语
- 为吸睛而编造数字、结果或身份

## 开头写法

前 3 行只完成一件事：给读者继续读的理由。

可用方式：

- 痛点场景：`剪一条 30 秒的视频，找素材、调色、配乐，一晚上没了。`
- 具体结论：`这篇只讲一件事：怎么把失败重试做成可控流程。`
- 真实结果：`我把原来半天的步骤压到了 20 分钟，但最难的不是调用 API。`
- 小故事：`上周接到一个需求，10 条视频的预算只够拍 2 条。`
- 反常识观点：`AI 没有让这件事更简单，它只是把难点换了位置。`

禁止用报幕、背景铺垫、学术定义或“今天给大家分享”开头。

## 作者调性与去 AI 味

### 独立开发者调性

默认像一个靠谱的工程师复盘自己做过的事：有人味但不油腻，有观点但不煽动，技术细节自然进入叙事。

- 用第一人称讲真实决策、结果和限制
- 口语负责降低门槛，事实负责建立可信度
- 主动承认工具不擅长什么
- 观点必须落在案例、代码、日志或数据上
- 推广内容写成使用和选择过程，不写成产品发布会

### 硬核技术调性

- 术语可以更精确，但首次出现仍需一句大白话解释
- 强调架构、权衡、边界和可复现证据
- 少用情绪和故事，不牺牲手机扫读体验

### 推广种草调性

- 场景和收益可以更突出，但不得夸大
- 先解决真实问题，再自然引出产品
- 用截图、前后对比、成本或时间数据支撑卖点
- 控制感叹号、emoji 和网感词密度

### 去 AI 味

完稿时执行：

1. 删除“此外”“值得注意的是”“综上所述”“在此基础上”等填充词
2. 打破连续的三段式排比和等长句
3. 删除“革命性的”“强大的”“令人惊叹的”等宣传形容词
4. 不写“业内人士认为”，要么给来源，要么表达为作者判断
5. 删除“未来可期”“相信会越来越好”等通用积极结尾
6. 中英文、中文与数字之间加空格，专有名词按官方写法
7. 控制破折号、引号和单行金句的密度

口语化可以有温度，但不能有段子味。优先使用准确表达：

| 避免 | 改为 |
|------|------|
| 模型犯傻、API 抽风 | 模型出错、API 不稳定 |
| 片子废了、直接崩 | 成片不可用、流程中断 |
| 搞了个玩意 | 做了一个工具 |
| 拉胯、逆天 | 表现不佳、超出预期 |

“踩了个坑”“不优雅，但实用”“费了不少时间”等自然工程表达可以保留。

## 手机阅读排版

完整布局、编辑器兼容和发布前预览规则读取 `references/wechat-layout.md`。本节只保留写作阶段的节奏约束。

- 每段尽量不超过 3-4 行
- 每 3-5 段设置一个小标题或视觉锚点
- 每节第一段优先给结论
- 加粗关键句，不整段加粗
- 连续纯文字不超过 5 段
- 小标题连起来应能复述主线

emoji 只在确实帮助识别章节时使用：全文不超过一半的小标题带 emoji，同一篇不超过 3 种。独立开发者调性优先少用。

## 常用结构模板

### 技术科普

痛点场景 → 大白话解释 → 原理 → 落地场景 → 边界和坑 → CTA

必须提供比喻或直观解释。涉及“怎么工作”时生成原理图；涉及完整流程时生成流程图。图片不可用时给出对应配图建议。

每个技术科普主体小节都要让读者多掌握一个判断动作。概念解释后至少补一个最小例子、一个可观察现象或一个自查问题；不要只停在“它是什么”和“它很重要”。

### 教程干货

目标 → 前置条件 → 操作步骤 → 每步结果 → 踩坑 → 完整清单 → CTA

每步必须有动作和预期结果。教程重点是让读者做成，不必展开无关原理。

步骤里必须写清成功信号和失败信号。没有验证方法的教程不算完成；如果某一步无法验证，就说明这是外部依赖或平台限制，并给出发布前检查项。

### 项目复盘

一句话结果 → 为什么做 → 关键决策 → 问题与解决 → 结果数据 → 下一步 → CTA

重点写取舍和失败处理，不写开发流水账。至少提供一个真实工程细节。

复盘里的每个主体小节都要沉淀一个可复用判断：为什么选这个方案、放弃了什么、下次遇到同类问题先查什么。只记录时间线、不交代判断依据的内容要删。

### 行业观察

具体观点 → 可核实实战案例 → 实践中的坑点 → 2-3 个论据 → 不成立的边界 → 对读者的实际意义 → 开放问题

观点必须可讨论，不能只做信息汇总。案例来自外部时明确出处，不改写成作者亲历。

行业观察不写成操作教程，但必须给读者判断方法。每个主体小节至少交付一个信号、一个边界、一个对比维度或一个迁移问题，让读者能判断自己所在行业、团队或项目是否适用。

### 产品推广

真实问题 → 旧方案的代价 → 实际使用过程 → 遇到的坑点与处理 → 产品带来的具体变化 → 证据 → 不适用场景 → CTA

价值先行，植入靠后。把功能翻译为读者能感知的时间、成本、质量或操作收益。

产品推广最多保留 3 个核心卖点，每个卖点都要有真实使用证据、对比条件或不适用边界。不能用“功能强大”“体验丝滑”替代具体变化。

## 资料研究与数据补全

用户只给主题、提纲或参考文档时，把这些内容视为研究起点，不是文章事实的全部来源。主动补齐文章兑现标题所需的真实信息。

### 最少追问原则

不要把“素材不足”变成默认追问。能由助手自主获取的信息，先自己获取；能通过降级写法避免事实风险的，先写有边界的草稿；只有缺失信息会改变文章立场、核心案例或事实真实性时，才追问用户。

优先自主获取：

- 当前对话里用户已经给出的目标、反馈、过程、坑点和验证方式
- 当前工作区的 README、示例、配置文件、测试命令、Git diff 和最近修改
- 用户提供的文档、链接、截图、日志和生成产物
- 官方文档、release note、issue、公开复盘和真实案例

### 主题扩展研究

用户给出的内容通常只是文章切口，不是文章边界。写作前必须判断读者要理解这个主题，还缺哪些背景和维度。尤其是产品、工具、平台、框架和行业观察类文章，不能只复述用户亲历。

产品或工具类文章必须主动建立“能力地图”：

| 维度 | 要查什么 | 写作时怎么用 |
|------|----------|--------------|
| 产品定位 | 官方如何定义产品，面向谁 | 防止把工具写错场景 |
| 入口与模式 | 该产品实际存在的网页端、桌面端、移动端、命令行、插件、工作模式或专业模式 | 帮读者判断该从哪里开始 |
| 核心能力 | 编码、调试、设计还原、自动化、文档、数据、研究、发布等 | 不只围绕用户说过的一项能力 |
| 扩展能力 | Skills、MCP、规则、命令、Hook、外部集成 | 写出可复用工作流，而不是单次体验 |
| 运行环境 | 本地、云端、沙箱、权限、文件访问边界 | 写清适用条件和安全边界 |
| 价格与限额 | 日限额、周限额、模型额度、排队、套餐、区域差异 | 涉及时必须查官方最新资料，不能凭印象写 |
| 典型场景 | 独立开发、团队协作、项目优化、从零开发、设计转代码 | 把用户案例放进更大的场景图 |
| 不适用场景 | 私有服务、密钥、内网依赖、缺少启动条件、需求不清 | 避免软文感和误导读者 |
| 最新状态 | 发布时间、功能是否更名、旧名称是否仍在文档中出现 | 防止写出过期名称或虚构使用时长 |

如果主题是某个具体产品，不得只用用户提供的体验写完整文章。至少使用官方资料补齐产品定位、入口、核心能力和限制；涉及价格、额度、模型、订阅、排队、发布状态等易变信息时，必须优先查官方文档。无法确认时，不写确定结论，只在发布前检查列为待确认项。

写 AI 编程工具、开发平台、效率工具或多入口产品时，默认补齐以下问题：

- 它到底是单一工具，还是由多个入口、模式或插件组成的产品生态？
- 编码能力之外，还有没有设计、文档、自动化、数据处理、工作流沉淀或外部集成？
- 它适合独立开发者的哪个环节：立项、设计、编码、审查、测试、发布、运营？
- 它有哪些使用成本：额度、排队、上下文管理、权限、安全、环境配置？
- 用户亲历能证明哪一点，官方资料能证明哪一点，哪些只能作为待验证判断？

文章中要把“用户亲历”放在一个具体案例位置，把“官方资料和外部信息”用于补全全貌。不能因为用户只讲了项目优化，就把产品写成只会项目优化。

### 多模式产品必须讲取舍

遇到多入口、多模式、多客户端或插件化产品时，不能只列功能清单。读者真正关心的是：为什么要拆开，拆开之后对我有什么好处，又多了什么成本。

必须回答：

- **为什么拆分**：不同任务需要不同上下文、工具权限、交互方式和产物形态
- **拆分的好处**：减少单一界面复杂度，让编码、设计、自动化、文档处理各自更聚焦
- **拆分的代价**：新用户需要理解入口差异，跨模式切换会增加学习成本
- **如果放在一起会怎样**：入口更简单，但界面和上下文容易变重，任务边界更模糊
- **怎么选择**：根据当前任务是写代码、审项目、做设计、跑自动化、查资料还是做汇报来选入口

推荐写法不是表格堆满功能，而是先给一句判断：

```text
这个产品把多个模式拆开，不是为了制造更多入口，而是为了让不同任务使用不同的上下文、权限和产物边界。
```

再给读者一个选择标准：

```text
先判断你当前任务的产物是什么：代码、设计稿、文档、数据结果、自动化流程，还是发布内容。产物不同，入口就不该强行相同。
```

追问必须满足两个条件：

1. 不问会导致正文核心事实可能写错
2. 这个信息无法从当前工作区、历史对话或公开资料中获取

追问时一次只问最关键的 1-3 个问题。不要为了把文章写得更漂亮，让用户补一长串背景材料。

### 先建立缺口清单

写作前检查是否缺少：

- 解释主题所需的权威定义或工作原理
- 能说明规模、变化或效果的真实数据
- 至少一个落地实践案例
- 实践中出现的具体问题、失败或限制
- 能帮助读者行动的步骤、清单或检查方法
- 涉及时效性的最新状态、版本和平台规则

只查与文章主线相关的缺口，不为显得“资料多”堆砌数据。

### 来源优先级

按以下顺序选择来源：

1. 官方文档、标准、政府或机构公开数据、公司财报和官方技术复盘
2. 原始研究论文、公开数据集、产品 release note、issue 和故障报告
3. 有采访、原始材料或明确数据来源的专业媒体
4. 能提供可复现步骤的技术社区文章

营销软文、无作者转载、搜索摘要、聚合页和没有方法说明的“行业报告”不能作为关键数据的唯一来源。

### 真实数据的使用规则

每个重要数字至少确认：

- **谁发布的**：来源主体
- **什么时候**：统计周期或发布日期
- **统计什么**：指标定义和单位
- **覆盖谁**：地区、样本、产品版本或使用场景
- **和什么比**：基线、对照组或前后条件

例如，不只写“效率提升 40%”，而要写清“某团队在什么任务、多少样本、什么时间范围内，相比哪种旧流程提升 40%”。缺少这些条件时，降低结论强度，不把局部结果推广成普遍规律。

### 外部案例如何写成真实实践

外部案例必须包含：主体、场景、采取的动作、可观察结果、遇到的坑点和来源。使用明确归属：

```text
某团队在 [场景] 中采用了 [做法]。公开复盘显示，他们遇到 [现象]，原因是 [原因]，随后通过 [处理] 完成验证。[来源名称，日期]
```

不能把它改写成“我试过”“我们踩过”。如果来源只披露结果，没有披露原因或处理过程，就只写已公开部分，不补造完整故事。

### 交叉核验与冲突处理

- 关键数据尽量找到原始来源；媒体报道只作为线索
- 两个来源数字冲突时，先检查统计时间、定义、样本和地区是否不同
- 无法解释冲突时，在正文中说明口径差异，不武断选择更吸睛的数字
- 时效性信息优先使用最新来源，并保留具体日期
- 医疗、法律、金融等高风险主题至少使用权威一手来源，不依赖社区经验

### 来源呈现

正文中在数据或案例首次出现处自然带出来源主体和日期，不在文末单独列"资料来源"板块。不要把检索过程或未采用的资料塞进正文。

## 真实实战与坑点

### 什么算真实实战

真实实战必须能追溯到至少一种素材：

- 用户明确讲述的亲身经历、操作步骤和结果
- 项目中的代码、配置、提交记录或 issue
- 真实运行日志、报错、测试结果、截图或导出产物
- 按上一节完成核验的外部案例；使用时明确出处，不能改写成作者亲历

行业常识、模型推断和网上常见说法不能写成“我试了”“我发现”“我踩过”。找不到亲历证据时，主动检索真实外部案例；仍然找不到才追问用户，不要用虚构细节填满文章。

一段合格的实战经验至少讲清：

1. **场景**：当时在做什么，使用什么项目、环境或约束
2. **动作**：实际尝试了什么，不写“做了一些优化”
3. **判断**：为什么这样选，放弃了什么方案
4. **结果**：看到了什么可观察变化
5. **证据**：对应代码、日志、截图、测试或用户原始陈述

不要只报一个漂亮结果。把关键条件和代价一起说清楚，读者才知道经验能否迁移到自己身上。

### 什么算真实坑点

坑点不是一句“这里容易踩坑”，而是一条完整的问题链：

```text
触发场景 → 可观察现象 → 根本原因 → 实际处理 → 验证结果 → 读者如何避免
```

写作时回答：

- **触发场景**：什么输入、配置、版本或操作触发了问题？
- **故障现象**：屏幕、日志、结果文件或用户体验具体发生了什么？
- **根本原因**：哪段代码、配置或机制导致？有什么证据支持？
- **实际处理**：改了什么，为什么这能解决问题？
- **验证结果**：重跑了什么，如何确认已经修好？
- **避免方法**：读者怎样提前检查或建立自动化防线？

如果真实过程里有“最初以为是什么、试过但没用的方法”，可以加入，它能体现判断过程；没有证据就不要为了戏剧性虚构失败尝试。

### 清晰、浅显的表达顺序

先说读者能看到的现象，再解释术语和底层原因：

```text
先说人话：导出的视频看起来正常，但上传后完全没有声音。
再说技术：检查后发现文件里没有音频流，不是播放器静音。
最后给办法：导出完成后用 ffprobe 检查是否存在 audio stream，没有就直接让任务失败。
```

避免抽象句：

| 不清楚 | 更清楚 |
|--------|--------|
| 效果不好 | 检索结果里有答案，但最终回复没有引用这段内容 |
| 配置有问题 | Release 配置仍读取了测试环境的 API 地址 |
| 后来优化了 | 把重试从无限循环改成最多 3 次，并记录最后一次错误 |
| 问题解决了 | 连续重跑 20 次没有复现，失败时也能返回明确错误 |

### 成稿硬检查

- [ ] 至少一段实战经验能对应到真实素材
- [ ] 至少一个坑点包含场景、现象、原因、处理和验证
- [ ] 数字带有测试条件，不把单次结果写成普遍结论
- [ ] 没把外部案例或行业常识写成作者亲历
- [ ] 外部案例写明主体和来源，数据写明时间、范围、口径和基线
- [ ] 术语第一次出现时有大白话解释
- [ ] 读者能据此复现检查步骤或提前避坑

任何一项不满足，都不能把文章标为“可直接发布”。先补素材，或在发布前检查中明确列为阻塞项。

## 输出状态与降级

生成完整文章前，先判断输出状态。这个判断用于约束写作，不要把内部检查表完整展示给用户。

### 可直接发布

同时满足以下条件：

- 每个主体小节都有自然呈现的可执行单元
- 整篇文章符合当前文章类型的完整落地链路
- 核心观点完成“结论、原因、证据、边界、读者怎么用”五层展开
- 实战、坑点、数据和外部案例都有可识别证据
- 长文每段都有功能，没有只负责铺垫、重复、转场或渲染情绪的空段

通过时，发布前检查只写真实待办；没有待办就写“无”。

### 需要补素材

主题方向明确，但缺少真实案例、日志、截图、数据、试用过程或验证结果时，不写完整成稿。先输出最少补充问题，问题要直指文章质量闸门。

但如果现有素材已经足够支撑“过程复盘、判断方法或工作流改进”，不要停在补素材清单。应先输出公众号式草稿，把缺少的 bug 详情、截图、数据或链接放到发布前检查。正文中用克制表达避开无法证明的结论。

优先问这类问题：

- 实际跑通过吗？输入是什么，输出是什么？
- 失败过哪一步？看到的报错、异常结果或截图是什么？
- 你为什么选择这个方案，放弃过哪些替代方案？
- 修复后怎么确认问题解决了？
- 有没有可以引用的官方文档、issue、release note 或公开案例？

### 方法论草稿

用户暂时没有真实素材，也不适合继续追问时，可以输出方法论草稿，但必须明确标注“当前是方法论草稿，不是可直接发布稿”。这类稿件只能写通用判断框架、资料来源和待验证假设，不得使用“我试过”“我踩过”“我们验证了”等亲历表达。

### 降级也要像公众号

降级不是把内部判断直接丢给用户。除非用户明确要诊断 Skill 表现，否则不要用以下内容开头：

- “当前状态：需要补素材”
- “还差 1 条信息”
- “我不能写成可直接发布稿”
- “以下是文章骨架”

更好的处理方式：

1. 正常输出标题、摘要和正文
2. 正文只写已有证据能支撑的内容
3. 对缺少证据的部分改成判断方法、流程建议或待验证边界
4. 在发布前检查中列出少量待补项

例如缺少具体 bug 单时，不要停止写作。可以把文章从“TRAE Work 修好了哪些 bug”改成“TRAE Work 帮我把项目优化流程收敛住”，正文重点写审查边界、bug 单汇总、避免无限循环和交叉验证。发布前检查再提示：发布前建议补 1 条真实 bug 单，增强可信度。

## 技术内容翻译

- 第一次出现术语时先说人话：`RAG（让 AI 先查资料再回答）`
- 能用 5-10 行关键代码说明的，不贴完整文件
- 解释“为什么这样设计”，不要逐行翻译代码
- 优先使用真实日志、报错、重试结果和前后对比
- 长代码更适合截图，但截图必须清晰并配文字说明
- 原理最多展开两层，除非目标读者是专业开发者

独立开发者文章至少加入一个“看起来不起眼，但不做就会失败”的真实工程细节，并按上一节的坑点链路讲清楚。这类细节比泛泛描述“调用了 API”更有价值。

## 产品推广规则

- 先交付一个真实问题的解决思路，再出现产品
- 不写“功能强大”，写清旧流程和新流程的差异
- 数据必须来自真实测试，并说明测试条件
- 主动说明不适用场景，避免制造错误预期
- 不冒充用户评价，不编造案例和销量
- 推广关系、赞助或利益相关应按实际情况披露

## 结尾与转化

结尾不重复总结全文，只完成一个动作：

- 涨粉：预告下一篇的具体价值
- 互动：提出读者能回答的具体问题
- 转化：给出低门槛、可执行的下一步
- 转发：说明这篇能帮哪类人解决什么问题

如果正文有“思考延伸”或“下一步”，先用 1-2 句回扣主题，再接 CTA，避免突然结束。

项目复盘、产品体验、行业观察和个人经验类文章结尾必须有“自我思考”，不能只停在工具推荐。教程干货和纯技术科普不强制第一人称反思，但必须保留判断边界和下一步建议。自我思考回答三个问题中的至少两个：

- 这次经历改变了作者哪个判断？
- 下次遇到同类任务，作者会先做哪件事？
- 作者现在仍然保留什么怀疑、限制或不确定性？

合格示例：

```text
这次之后，我对 AI 编程工具的期待反而变低了：它不应该替我无限优化项目，而应该帮我把审查边界固定下来。

下次再做项目优化，我会先让它只产出 bug 单，不直接修。等范围收敛后，再进入修复和交叉验证。
```

不合格写法：

```text
总的来说，TRAE Work 是一款值得尝试的工具，建议大家根据自己的情况选择。
```

外链需要遵循用户当前公众号能力和平台规则。不要默认声称某一种入口永远可用；把目标链接列入发布前检查，并提示发布前实测。

转发文案要像真人说话，指出具体内容和适合谁看。避免“干货满满”“建议收藏”“转发给所有需要的人”等空话。

## 合规检查

- 医疗、金融、教育等高风险领域不做保证性承诺
- 不使用无法证明的“最”“第一”“国家级”等表达
- 数据、研究、报道和他人观点标明可核实来源
- 不编造用户案例、对话、评价或实验结果
- 图片使用用户授权、可商用或自行生成的素材
- 不设计强迫、欺骗或利益诱导式分享文案
- 涉及时效性平台规则时，在发布前重新核实
