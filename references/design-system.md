# 视觉设计系统（全局基础约束）

正文图片 reference（`cartoon-illustrations.md` / `flowchart-style.md`）继承本文的全局视觉规范。封面图只继承共享色板和 IP 定义，背景、构图和线条按 `cover-image.md` 的风格覆盖规则执行。

**使用方式：** 生成正文图片前，把本文的「全局风格 token」放在 prompt 开头，再叠加对应 reference 的专属风格词。生成封面时不要直接粘贴正文 token，只读取共享颜色后使用封面模板。

## 全局配色系统

正文图片统一从以下色板取色。封面可按 `cover-image.md` 的风格使用深色背景，但强调色保持一致。

| 色号 | 名称 | 用途 |
|------|------|------|
| `#F4845F` | 暖橙（主色） | 核心元素、主角节点、视觉锚点 |
| `#FDE68A` | 浅黄（辅色） | 次要元素、背景节点、高亮区 |
| `#FAFAF8` | 暖白（背景） | 所有图片的统一背景色 |
| `#1A1A1A` | 墨黑（线条） | 手绘轮廓线、文字标注 |
| `#6B7280` | 中灰（弱化） | 次级标注、引导线、说明文字 |
| `#0F172A` | 深海蓝 | 深色封面背景 |

**使用原则：**
- 单张图内不超过 3 种颜色（背景色不计）
- 暖橙永远是视觉重心，浅黄永远是支撑，不要反过来
- 正文图片统一使用暖白底（`#FAFAF8`）；封面背景由所选封面风格决定

## 线条与笔触规范

| 参数 | 规范 |
|------|------|
| 线条风格 | 手绘抖动感，有轻微不规则（slight wobble），不是光滑矢量线 |
| 线条粗细 | 主体轮廓中等粗细，标注引导线细 |
| 填充方式 | 平涂（flat fill），不用渐变、不用纹理 |
| 阴影 | 不用投影，可用极轻的线条内阴影增加手绘感 |

## 留白规范

| 图片类型 | 主体占画面比例 | 说明 |
|---------|-------------|------|
| 叙事插图（橘猫） | 40%–60% | 角色居中偏上，下方留白 |
| 流程图 / 架构图 | 50%–70% | 图表居中，四边留均匀边距 |
| 原理示意图 | 40%–60% | 意象居中，标注散布周围 |
| 封面图 | 30%–50% | 主视觉居中，两侧各留 20% 安全边距 |

## 文字标注规范

- 标注语言：**中文优先**，专有名词（API 名、技术栈）保留英文原文
- 标注字数：单条标注不超过 **6 个汉字**，保证手机屏幕可读
- 标注风格：handwritten-style，不用印刷体感的正式字体
- 引导线：细虚线或细实线，不用粗箭头指向
- 标注密度：单图标注不超过 **5 处**，超过则拆图

## 图片内中文文字规范

图片内的文字标注**必须使用中文**（专有名词如 RAG、API、Token 等保留英文原文）。图像模型渲染中文有一定出错率，通过以下 prompt 工程手段将错误率降到可接受范围，**无需任何外部工具或脚本依赖**。

### 提高中文渲染准确率的四条规则

**① 控制每张图的中文标注总量：≤ 4 条**

标注越少，模型出错概率越低。超过 4 条标注的图，优先考虑拆成两张。

**② 每条标注字数控制在 2-4 个汉字**

字越短，渲染越准。避免长句标注：

| ❌ 太长 | ✅ 精简 |
|--------|--------|
| 「模型生成最终回答」 | 「生成回答」 |
| 「从知识库检索相关内容」 | 「搜知识库」 |
| 「用户输入的问题」 | 「用户提问」 |

**③ 在 prompt 里逐字写出节点文字，用引号括起来**

不要让模型自己决定写什么，明确告诉它每个节点/标注的具体文字：

```text
# ✅ 正确写法 — 逐字指定
Node 1 label: "用户提问"
Node 2 label: "搜文档"
Node 3 label: "生成回答"

# ❌ 错误写法 — 让模型自由发挥
Nodes showing the RAG process with Chinese labels
```

**④ 专有名词保留英文，不强行翻译**

RAG、API、Token、Prompt、Embedding 这类在中文语境里约定俗成用英文的词，直接写英文，图像模型渲染英文字母几乎不出错：

```text
Node label: "RAG 流程"   ✅  (而不是「检索增强生成流程」)
Node label: "API 调用"   ✅  (而不是「接口调用」)
```

### 生成后验证

图片生成后，检查所有中文标注是否正确。如有错字，在 prompt 里针对出错的标注单独强调：

```text
IMPORTANT: The label on node 2 must read exactly "搜文档" — 
these are the specific characters: 搜(search) 文(text) 档(file).
```

## 宽高比规范

| 图片类型 | 宽高比 | 原因 |
|---------|-------|------|
| 叙事插图（橘猫） | 16:9 | 横幅宽图，适合正文内嵌 |
| 流程图 / 架构图 / 原理图 | 4:3 | 竖屏手机上信息密度更合理 |
| 封面图 | 16:9（生成）→ 2.35:1（裁切） | 图像模型用 16:9，发布时按平台裁切 |

## 配图决策原则

**核心问题只有一个：把这段文字换成图，读者能不能更快、更容易看懂？**

能 → 配图。不能 → 不配。

不要等「符合某个场景」才配图。通读每一段正文时，主动问自己：

> 这段内容有没有关系、结构、流向、对比、因果——这些用图比用文字表达得更清楚？

有的话就配。没有的话就不配。

**配图价值自检（每段过一遍）：**

1. 这段在讲「步骤/流程」→ 画出来比读出来快吗？ → 快就配
2. 这段在讲「A 和 B 的区别」→ 并排对比比文字列举清楚吗？ → 清楚就配
3. 这段在讲「为什么/怎么工作」→ 一个意象或示意图能让读者秒懂吗？ → 能就配
4. 这段在讲「N 种方法」→ 每种方法能不能各自用一张图表达清楚？ → 能就配
5. 这段有核心比喻 → 把比喻画出来比读文字更直观吗？ → 是就配

**不配图的情况：**
- 纯叙述/过渡段落，没有可视化的结构或关系
- 概念一句大白话说清，图不会增加理解速度
- 步骤 ≤ 2 且没有分支，文字 + emoji 序号足够

## 常见场景的配图类型参考

以下是高频出现的内容模式和对应配图类型，**不是穷举，只是参考**。只要符合「配图能让读者更快看懂」这个原则，不在表里的场景也应该配图。

| 内容模式 | 配图类型 | 走哪个 reference |
|---------|---------|----------------|
| 步骤流程（≥ 3 步） | 线性流程图（A） | `flowchart-style.md` |
| 条件分支判断 | 分支决策图（B） | `flowchart-style.md` |
| 多模块架构/数据流 | 架构关系图（C） | `flowchart-style.md` |
| 技术原理/实现逻辑/为什么会这样 | 原理示意图（D） | `flowchart-style.md` |
| 技术对比/方法对比/适合 vs 不适合 | 对比清单图（E） | `flowchart-style.md` |
| 并列解法（每种有独立原理） | 每种各配一张流程图/原理图 | `flowchart-style.md` |
| 效果前后对比 | 橘猫对比插图 | `cartoon-illustrations.md` |
| 核心比喻（承载概念的生活化意象） | 橘猫叙事插图或原理示意图（D） | 见下方说明 |
| 开头痛点/踩坑/完成庆祝（有信息价值时） | 橘猫叙事插图 | `cartoon-illustrations.md` |

**情绪节点额外判断：** 开头痛点、踩坑等情绪场景，配图前先过「可视化价值自检」——图能传达具体信息、且不与正文重叠，才配。纯装饰性的情绪图不配。

**橘猫功能性测试：** 无论叙事图、流程图还是封面，只要加入橘猫，生成前必须写出一句角色说明：

```text
橘猫代表 [具体人物或状态]，正在 [与主题直接相关的动作]，帮助读者理解 [关系、流程、对比或情绪变化]。
```

三项有任何一项说不清，就不加橘猫。禁止把橘猫作为背景、角落贴纸、品牌水印或与主题无关的可爱装饰。

**比喻类选型：**
- 纯情感性比喻 → 橘猫叙事插图
- 机制性比喻 + 需要标注 2 个以上技术概念 → 原理示意图（D）

## 各图片类型的风格叠加关系

每次生成图片时，prompt 按以下顺序拼接三段内容（以下为示意，实际生成时用具体内容替换，不要把尖括号或中文占位描述输出进 prompt）：

1. **全局风格描述**：从本文件「全局视觉 Token」代码块第二行起内联，去掉方括号标识行
2. **该图片类型的专属风格词**：从对应 reference（`cartoon-illustrations.md` / `flowchart-style.md` / `cover-image.md`）中取出专属风格描述
3. **具体场景/内容描述**：当前图片的画面内容、节点文字、人物动作等

**专属风格词的作用是「覆盖或补充」全局 token，不是替换。** 如果专属风格词与全局 token 有冲突，以专属风格词为准（例如封面图使用深色底，覆盖全局的白底规定）。

## 跨图一致性检查

同一篇文章里有多张图时，完成所有图片生成后走一遍：

- [ ] 所有正文插图（叙事图 + 流程图 + 原理图）背景一致：白底或极浅暖灰
- [ ] 主色暖橙在每张图里都是视觉重心
- [ ] 线条都有手绘抖动感，没有一张是光滑矢量风格
- [ ] 文字标注字数都在 6 字以内，手机预览清晰
- [ ] 封面图与正文图的风格差异是「有意为之」（深色 vs 白底），不是风格失控
- [ ] 橘猫 IP 在叙事插图间保持一致（圆润身体、点状眼睛、橘色虎斑）
- [ ] 每只橘猫都有明确角色和主题动作，不是背景、角落贴纸或无意义装饰
