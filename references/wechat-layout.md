# 微信公众号正文布局规范

生成完整文章时读取本文。目标是输出复制到公众号编辑器后仍然层级清楚、适合手机扫读的语义排版，不依赖复杂 HTML、CSS 或第三方编辑器私有样式。

本文的字数和数量是移动阅读建议，不是微信平台的强制技术限制。用户提供品牌模板时，优先保留品牌规范，但仍执行兼容性和扫读检查。

## 默认版式骨架

正文按以下节奏组织，不把输出格式说明写进文章：

```text
2-3 段短开头

核心结论或承诺

一级内容模块
结论句 → 解释 → 实战证据 → 图片或列表 → 小结/过渡

坑点模块
场景 → 现象 → 原因 → 处理 → 验证 → 避免方法

可收藏清单

结尾收束 + 单一 CTA

资料来源（使用外部资料时）
```

不要在正文开头放目录。小标题本身就是移动端的导航。

## 段落与留白

- 一段只表达一个意思，通常 1-3 句
- 长段优先按“结论 / 证据 / 解释”拆开，不为拆段切断完整句子
- 段落、标题、图片、引用块和列表之间保留一个空行
- 不连续使用多个空行制造虚假留白
- 连续纯文字不超过 4 段；之后加入小标题、列表、引用、图片或短结论句
- 单句重点可以独立成段，但不要连续制造“金句墙”

判断标准：手机首屏不能被一整块文字填满；快速滑动时仍能看出内容节奏。

## 标题与重点

### 标题层级

- 正文只使用二级和三级标题，不跳级
- 二级标题负责主线，三级标题只用于同一模块内确有并列子问题时
- 标题直接给信息，如“坑一：图片规则互相冲突”，避免“问题来了”“再聊几句”
- 标题尽量保持一行可读，不把完整结论塞进标题
- 全文不超过一半的标题使用 emoji，同一篇不超过 3 种；默认少用

### 加粗

- 每个小节通常突出 1-2 个短句或关键词
- 加粗承担结论、数字、关键动作或风险提示
- 不整段加粗，不同时混用加粗、斜体、下划线和多种颜色
- 相邻两段不要都以加粗句开头，避免视觉疲劳

## 引用、列表和分隔线

### 引用块

只用于以下内容：

- 原始资料中的关键原话
- 一句话定义或边界
- 需要与作者叙述区分的外部观点

引用块保持 1-3 句，并紧跟来源。不要把普通总结、配图说明和整段正文都做成引用块。

### 列表

- 适合步骤、检查项、方案对比和并列结论
- 一组通常 3-7 项；超过 7 项时按主题拆组
- 每项优先使用“**关键词**：解释”的结构
- 避免超过一层的嵌套列表；复杂层级改成多个小标题
- 步骤必须使用数字顺序，普通并列项使用无序列表

### 分隔线

- 只在开头、正文、结尾等大区块切换时使用
- 不在每个二级标题前后都放分隔线
- 连续短小节依靠标题和留白区分，不堆装饰符号

## 图片与图注

- 图片紧跟在首次提及它的段落之后，不把所有图片集中到文末
- 图片前用一句话说明“接下来看什么”，图片后补一句结论或图注
- 图注说明图片展示的事实、对比或结论，不重复 alt 文本
- 两张图片不要无说明地连续出现；对比图优先合成一张或明确标注“前 / 后”
- 图片内文字必须在手机预览中可读，关键信息不能只存在于图片里
- 正文图片遵循 `design-system.md`；封面遵循 `cover-image.md`

纯文字降级模式下，配图建议放在对应段落位置，不集中列成独立“配图方案”。

## 表格、代码和链接降级

### 表格

公众号手机页面不适合宽表格。正文默认不输出 Markdown 表格。

- 2-3 个维度的简单对比：改为“方案 A / 方案 B”分组列表
- 多列数据：提炼 2-4 个关键数字，剩余内容生成对比图或长图
- 必须保留表格时：控制列数和文字长度，并在发布前用手机预览检查

### 代码

- 只保留解释核心判断所需的 5-10 行
- 单行过长时先删去无关参数，避免手机横向滚动
- 长代码改为关键片段 + 仓库链接，或生成清晰截图
- 代码前说明它解决什么问题，代码后解释为什么这样写
- 命令、字段名和文件名使用行内代码，不把普通中文塞进代码块

### 链接

- 正文不堆裸链接，先写来源主体和结论
- 引用外部资料时，在正文首次出现处自然带出来源主体和日期，不在文末单独列"资料来源"板块
- 行动链接放在 CTA 或发布前检查指定的位置
- 不使用 Markdown 脚注语法，避免复制到编辑器后丢失引用关系

## 实战与坑点布局

实战经验使用“动作推进”布局：

```markdown
## 我实际怎么做的

先交代场景和限制。

**关键判断：为什么选择这个方案。**

给出代码、日志、数据或截图证据。

最后说明结果和适用边界。
```

坑点使用固定的信息标签，方便扫读，但不要写成僵硬表格：

```markdown
## 坑点：一句话说清问题

**触发场景：** 什么条件下发生。

**看到的现象：** 用户或日志具体看到什么。

**根本原因：** 用大白话解释，再补技术术语。

**实际处理：** 改了什么，为什么有效。

**验证结果：** 如何确认已经修好。

**怎么避免：** 给读者一个检查或自动化办法。
```

如果文章有多个坑点，每个坑点保持相同标签顺序，让读者形成稳定预期。

## 编辑器兼容规则

- 默认输出标准标题、段落、加粗、引用、列表、代码块和图片，不输出复杂 HTML/CSS
- 禁止 Mermaid、轮播、标签页、折叠面板和依赖 JavaScript 的组件
- 不依赖自定义字体、绝对定位、浮动、多栏布局或背景图承载正文
- 不用连续空格、特殊 Unicode 空白或大量全角空格控制缩进
- 不用颜色作为唯一的信息区分方式，确保样式丢失后仍能读懂
- 复制到公众号编辑器后必须做手机预览；不能只看 Markdown 渲染结果

## 发布前布局检查

- [ ] 开头 2-3 段内已经进入主题
- [ ] 只看二级标题能复述文章主线
- [ ] 没有填满手机首屏的文字墙
- [ ] 每节加粗克制，没有整段加粗
- [ ] 列表没有多层嵌套，步骤顺序明确
- [ ] 正文没有未经手机预览的宽表格
- [ ] 代码片段短且单行可读，长代码已降级
- [ ] 图片位置、图注和正文互相解释，不连续堆图
- [ ] 外部资料在正文就近归属，并在文末统一列出
- [ ] CTA 与正文之间有清楚收束，不突然插入链接
- [ ] 复制到公众号编辑器后完成一次手机预览
