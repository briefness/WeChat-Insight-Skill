# 多轮修改模式工作流

当用户对已生成的文章提出修改要求时，进入本模式。目标是：**精准修改、最小改动、保持一致性、修改后完整自检**。

## 触发场景

以下用户输入自动进入修改模式：

- "改一下第二段"
- "标题换个风格"
- "把 CTA 改成关注公众号"
- "调性再硬核一点"
- "这个例子换一下"
- "太长了，缩到一半"
- 所有针对已输出内容的局部修改指令

## 核心原则

### 1. 最小改动原则

只改用户明确要求修改的部分，不借机重写全文或调整未提及的段落。

**反例：** 用户说"把标题改得更吸引人"，结果连正文结构也一起改了。

### 2. 一致性维护

修改后必须检查：被修改部分与全文其他部分是否仍然自洽。

- 改了标题 → 检查摘要、正文开头、CTA 是否与新标题一致
- 改了论点 → 检查前后段落的逻辑衔接是否仍然成立
- 改了调性 → 检查全文语气是否统一，没有半截旧调性残留
- 改了数据 → 检查前后引用同一数据的地方是否同步更新

### 3. 修改留痕

每次修改后，简要说明改了什么、为什么这么改。不需要长篇大论，但要让用户一眼看出变化。

## 修改工作流（5 步）

### Step 1：定位修改范围

先明确用户要改的是哪一部分：

| 修改类型 | 范围判断 |
|---------|---------|
| 标题修改 | 只动备选标题列表，可能需要同步更新摘要和封面 |
| 摘要修改 | 只动摘要，检查字数是否在 40-80 字区间 |
| 开头修改 | 只动前 1-3 段，检查首屏是否仍有钩子 |
| 某一节修改 | 只动指定小节，检查前后衔接 |
| 调性修改 | 全文语气调整，但结构不变 |
| 长度调整 | 按比例增减，优先删/加过渡和例证，不碰核心论点 |
| CTA 修改 | 只动结尾，检查收束句与 CTA 的衔接 |
| 配图修改 | 只动对应位置的图片或配图建议 |

不确定时，用一句话向用户确认范围，不要猜。

### Step 2：执行精准修改

- 未被要求修改的部分，**逐字保留**，不做"顺手优化"
- 修改部分遵循原有写作规则（读取 `writing-playbook.md` 和 `wechat-layout.md`）
- 如果是调性/风格类修改，先确定目标调性的规则，再逐段对照调整
- 如果是结构性修改（加一节、合并两段），先确认修改后的结构是否仍然符合文章类型模板

### Step 3：一致性回扫

修改完成后，必须做以下检查（只扫与修改相关的上下游）：

**标题修改后必查：**
- [ ] 摘要是否仍能概括新标题的核心承诺
- [ ] 正文开头是否能兑现新标题
- [ ] CTA 与新主题是否匹配

**正文修改后必查：**
- [ ] 修改段与上一段的衔接是否自然
- [ ] 修改段与下一段的过渡是否顺畅
- [ ] 小标题连起来的主线是否仍然成立
- [ ] 修改涉及的事实、数据、案例是否有来源支撑

**调性修改后必查：**
- [ ] 全文语气统一，没有突兀的旧调性句子
- [ ] 专业术语密度符合目标读者
- [ ] 去 AI 味检查仍然通过

**长度调整后必查：**
- [ ] 核心论点没有被删掉
- [ ] 真实实战和坑点仍然完整
- [ ] 传播抓手（粘性/流通/扩散/易读）仍然存在

### Step 4：完整发布前检查

**无论是新建还是修改，只要输出完整正文，就必须跑完 `SKILL.md` 第 6 步的全部自检。** 不允许因为"只是改了一点"而跳过。

坑点结构必须使用 `wechat-layout.md` 规定的信息标签格式。

### Step 5：输出修改说明

在输出末尾（发布前检查之后）增加一段"本次修改说明"，用 1-3 条短句说明：

```text
## 本次修改说明

- 修改了备选标题：将推荐标题从 X 改为 Y，更突出 Z
- 调整了第三节的案例：替换为更贴近读者场景的 A 案例
- 缩短了全文约 200 字：精简了背景铺垫，保留核心论点和坑点
```

如果只是微调（改几个字、换个说法），修改说明可以更短，但不能省略。

## 常见修改场景的处理规则

### 场景 1："标题不够吸引人，再改一版"

- 保留原有 5 个标题中还不错的 2-3 个，补充新的 2-3 个
- 重新判断哪个最适合作为"推荐"
- 同步检查摘要是否需要调整
- 修改说明列出：新增了哪几个、删掉了哪几个、推荐为什么换了

### 场景 2："这段太技术了，读者看不懂"

- 先判断：是术语没解释，还是整个概念需要降维
- 优先加一句大白话解释，而不是删掉技术内容
- 如果整个段落都需要改写，保留核心结论，用生活化比喻重写
- 检查前后段落的衔接：前面抛出的问题、后面的结论是否仍然对应

### 场景 3："太短了，再丰富一点"

- 优先补充：真实案例、具体数据、操作细节、坑点的更多维度
- 不补充：宏大背景、泛泛而谈的意义、重复的论点
- 补充后检查：信息密度有没有下降，有没有注水感

### 场景 4："太长了，精简一下"

- 优先删：过渡句、重复强调、背景铺垫、次要案例
- 不删：核心论点、真实实战、坑点链路、CTA
- 精简后检查：小标题连起来是否还能看懂主线
- 如果精简后实战或坑点被删掉了，主动告知用户"为了控制篇幅，去掉了 XX 案例，需要加回来吗？"

### 场景 5："换个例子/案例"

- 新案例必须同样满足真实性要求：有来源、有细节、有结果
- 替换后检查：案例引出的结论是否仍然成立
- 如果新旧案例结论略有不同，同步调整结论句

## 修改轮次的边界

- **不做无限迭代**：如果同一部分被修改超过 3 轮仍然不满意，停下来问用户："你想要的感觉能不能更具体描述一下？或者给我一个你觉得好的参考例子？"
- **不做方向漂移**：如果用户的修改要求本质上是换了一个文章主题（不是局部调整，是核心论点变了），建议用户"这相当于重写一篇，需要重新走一遍素材核实和结构设计吗？"
- **保留用户确认过的部分**：用户明确说过"这段没问题""这个标题可以"的部分，后续修改不要动

## 修改模式下的输出格式

输出顺序与新建时一致，只在末尾增加"本次修改说明"：

1. 使用假设（如果参数变了才更新，没变就沿用）
2. 备选标题（如果修改了标题）
3. 摘要（如果修改了摘要）
4. 正文（完整正文，不是 diff）
5. 转发文案（如果修改了）
6. 发布前检查（完整检查，不是只列新的）
7. **本次修改说明**（新增，1-3 条）

> ⚠️ **注意：** 正文永远输出完整版本，不输出"只改了 XX 段，其余不变"。用户需要的是可以直接复制使用的完整成稿。
