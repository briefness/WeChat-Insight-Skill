# 示例输出：一次真实的 Skill 优化复盘

> **使用假设**：项目复盘；目标读者为使用 AI Skill 的开发者；核心目标为经验分享 + 互动；采用独立开发者调性。
>
> 本示例基于当前仓库的真实修改、Git 基线文件和本地校验结果。由于图片工具未参与本次优化，示例采用纯文字降级模式。

## 摘要

一个写作 Skill 规则越多，不一定越稳定。这次我把入口文件从 656 行压到 178 行，同时处理了图片生成和橘猫使用规则的冲突。过程里最值得记录的，不是删了多少字，而是怎样让入口、reference、README 和示例只说同一种话。

## 正文

我原本只是想让这个公众号写作 Skill 稳定输出完整成稿。

真正打开文件后，问题比“提示词太长”更具体：**同一件事在不同文件里有不同答案。** 模型读到哪一段，最后行为就可能偏向哪一段。

这次优化有真实的前后数据：Git 基线里的 `SKILL.md` 是 656 行，当前版本是 178 行。详细写作规则没有被直接删掉，而是移进了 6 个按需读取的 reference。

## 先把入口变成路由器

旧版入口同时装着标题、语气、排版、传播、图片、封面和合规规则。内容很全，但每次触发都要一起加载。

这次我保留了真正决定执行的部分：输入判断、工作流、输出契约、文章类型路由和最终检查。标题、调性、结构模板、CTA 等细则移到了 `writing-playbook.md`。

**入口负责告诉模型“现在该做什么”，reference 负责告诉模型“具体怎么做”。** 这个边界定下来之后，删减就不再是单纯压缩文字。

## 坑一：图片到底必须生成，还是可以留占位？

**触发场景：** 我在检查“可直接发布”的输出契约时，同时对照了旧版入口、插图 reference 和示例文章。

**看到的现象：** 旧版入口要求配图必须实际生成，不能留占位符；插图 reference 又说写草稿时可以保留配图位；示例里放着文字占位，却声称图片已经自动生成。

换成大白话就是：**三个文件对“文章交付到什么程度”说法不一致。** 这不是文案问题，而是执行分支没有定义清楚。

**根本原因：** 图片规则被复制到了多个文件，但没有一个统一的能力判断：当前环境到底有没有图片工具，用户到底要完整图文还是纯文字稿。

**实际处理：** 我把流程明确分成两种模式：有工具时实际生成并嵌入图片；工具不可用或用户只要文字时，输出包含图片类型、画面、标注和比例的配图建议，同时明确说明图片尚未生成。

**验证结果：** 入口、README、示例和三个图片 reference 现在使用同一套模式名称；扫描旧的“图片已经生成”说法后没有残留。

**怎么避免：** 同一个能力如果出现两种合法结果，就在入口定义分支和降级条件，不要让每个 reference 自己猜。

## 坑二：橘猫有时是角色，有时只是贴纸

**触发场景：** 我继续检查图片规范，希望同一只橘猫能形成稳定的视觉 IP。

**看到的现象：** 旧版入口说流程图和封面不加橘猫；流程图 reference 允许在部分节点旁加猫；封面模板又把橘猫设成固定主视觉。

如果只把规则统一成“所有图都加猫”，问题仍然没有解决。流程图里多一只站在角落的猫，只会遮挡信息。

**根本原因：** 规则按图片类型决定“加不加猫”，却没有判断橘猫在画面里承担什么功能。

**实际处理：** 新规则要求生成前先回答三个问题：橘猫代表谁？正在做什么？它怎样帮助理解主题？答不出来就不加。背景、角落贴纸、品牌水印和单纯卖萌都不算合理用途。

**验证结果：** 叙事图、流程图、封面、设计系统和示例都加入了同一项功能性检查。比如这篇的封面里，橘猫不是站在标题旁边，而是在把冲突规则整理成分类 reference，它的动作就是文章主题。

**怎么避免：** 视觉 IP 不是每张图都出现，而是每次出现都有事情可做。

## 我最后用什么确认它真的改好了

我没有只看文件行数。最后实际执行了四类检查：

1. `git diff --check`：检查空白错误和补丁格式
2. YAML 解析：确认 frontmatter 只有 `name` 和 `description`
3. 相对链接扫描：确认 README 和 reference 链接都能找到文件
4. 旧规则关键词扫描：确认冲突颜色、旧标题选择流程和装饰性橘猫指令没有残留

这里还有一个小插曲：官方 `quick_validate.py` 在当前环境缺少 `PyYAML`，不能直接运行。我没有把“脚本启动失败”写成“Skill 校验失败”，而是用系统自带的 Ruby YAML 解析器完成等价的 frontmatter 检查，并在交付说明里保留这个限制。

**验证工具失败，和被验证对象失败，是两件不同的事。** 这也是这次最值得留下的实战细节。

## 一份可以复用的检查清单

- 入口是否只保留决策和路由，而不是塞满所有细节？
- 同一条规则是否在入口、reference、README 和示例中保持一致？
- 能力不可用时，有没有明确的降级结果？
- 示例展示的行为，是否真的是规则要求的行为？
- 校验失败时，分清是工具环境问题还是 Skill 本身有问题了吗？

这次重构让我更确定一件事：**Skill 的质量不取决于写了多少规则，而取决于每条规则有没有唯一、清晰、可验证的执行路径。**
