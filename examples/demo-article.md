# 示例输出：一次真实的 Skill 优化复盘

> **使用假设**：项目复盘；目标读者为使用 AI Skill 的开发者；核心目标为经验分享 + 互动；采用独立开发者调性。
>
> 本示例基于当前仓库的真实修改、Git 基线文件和本地校验结果。由于图片工具未参与本次优化，示例采用纯文字降级模式。

## 备选标题（5 个）

1. **推荐：我把一个公众号 Skill 从“规则很多”改到“路径清楚”**

3. 一个公众号写作 Skill 的真实优化记录：怎么发现、修掉、验证规则冲突

5. AI Skill 怎么越写越稳：这次重构留下的检查方法

## 摘要

一个写作 Skill 规则越多，不一定越稳定。这次我处理的不是“提示词太短”，而是入口、reference、README 和示例之间的规则不一致。最后留下的经验很具体：入口只做决策，细则放进 reference，冲突规则必须有降级分支和验证方法。

## 正文

我原本只是想让这个公众号写作 Skill 稳定输出完整成稿。

真正打开文件后，问题比“提示词不够强”更具体：**同一件事在不同文件里有不同答案。** 模型读到哪一段，最后行为就可能偏向哪一段。

这次优化有一个清楚的目标：让 Skill 在遇到配图、封面、示例和发布规则时，不再靠模型临场猜，而是沿着唯一的执行路径走。

> 【配图建议：封面图；橘猫代表正在整理 Skill 的开发者，把一摞互相冲突的规则整理成“入口”“写作”“图片”“发布”四个文件夹；标题突出“从规则很多到路径清楚”；16:9 生成后裁切为 2.35:1】

## 先定目标：不是压缩，而是让规则各归各位

这次修改一开始很容易走偏：看到入口文件很长，就想把它压短。

但单纯压缩不是目标。入口短了，如果 reference 里还保留冲突说法，模型照样会被带偏。真正要解决的是：**每条规则只能有一个归属，每个分支都要知道什么时候触发。**

我最后采用的划分很简单：

| 位置 | 负责什么 | 不负责什么 |
|------|----------|------------|
| `SKILL.md` | 触发条件、执行顺序、输出契约、硬性检查 | 不展开标题、语气和图片细节 |
| `writing-playbook.md` | 标题、结构、调性、真实案例和合规细则 | 不决定发布脚本怎么运行 |
| 图片 reference | 视觉系统、封面、插图和流程图规则 | 不改正文写作目标 |
| `README.md` | 面向使用者解释能力和边界 | 不承载内部执行细则 |

判断一个 Skill 入口是不是太重，可以先问三件事：这条规则是否决定“下一步做什么”？是否每次触发都要读？是否会和其他 reference 重复？三个问题只要有两个是否定，就应该移出入口。

## 再改结构：入口变成路由器

旧版入口同时装着标题、语气、排版、传播、图片、封面和合规规则。内容很全，但每次触发都要一起加载。

我保留了真正决定执行的部分：输入判断、工作流、输出契约、文章类型路由和最终检查。标题、调性、结构模板、CTA 等细则移到了 `writing-playbook.md`。

这个取舍不是为了“少写规则”。相反，细则还在，只是换了位置。入口负责告诉模型“现在该读哪份规则”，reference 负责告诉模型“具体怎么做”。

如果你也在拆 Skill，可以按这个顺序操作：

1. 先标出每条规则的动词：判断、生成、检查、发布、降级
2. 判断类和输出契约留在入口
3. 生成类细节移到对应 reference
4. 检查类只保留硬门槛，细项放到 reference
5. 发布类只保留触发条件和失败处理入口

这样做的验证标准也很直接：不看“删了多少行”，而是看同一个问题在项目里还会不会出现两个答案。

## 坑一：图片到底必须生成，还是可以留占位？

**触发场景：** 我在检查“可直接发布”的输出契约时，同时对照了旧版入口、插图 reference 和示例文章。

**看到的现象：** 旧版入口要求配图必须实际生成，不能留占位符；插图 reference 又说写草稿时可以保留配图位；示例里放着文字占位，却声称图片已经自动生成。

换成大白话就是：**三个文件对“文章交付到什么程度”说法不一致。** 这不是文案问题，而是执行分支没有定义清楚。

**根本原因：** 图片规则被复制到了多个文件，但没有一个统一的能力判断：当前环境到底有没有图片工具，用户到底要完整图文还是纯文字稿。

**实际处理：** 我把流程明确分成两种模式：有工具时实际生成并嵌入图片；工具不可用或用户只要文字时，输出包含图片类型、画面、标注和比例的配图建议，同时明确说明图片尚未生成。

**验证结果：** 入口、README、示例和三个图片 reference 现在使用同一套模式名称；扫描旧的“图片已经生成”说法后没有残留。

**怎么避免：** 同一个能力如果出现两种合法结果，就在入口定义分支和降级条件，不要让每个 reference 自己猜。

这类冲突可以用一张小表快速定位：

| 检查项 | 问题 | 应该写在哪里 |
|--------|------|--------------|
| 能力是否可用 | 当前环境有没有图片工具 | 入口 |
| 用户是否要求图片 | 完整图文还是纯文字 | 入口 |
| 生成什么图 | 封面、流程图、叙事图 | 图片 reference |
| 生成失败怎么办 | 降级为配图建议还是停止 | 入口 + 对应 reference |

> 【配图建议：分支流程图；橘猫站在“是否有图片工具”的判断节点旁，把流程分成“完整图文模式”和“纯文字降级模式”；标注“先判断能力，再决定交付形态”；4:3】

## 坑二：橘猫有时是角色，有时只是贴纸

**触发场景：** 我继续检查图片规范，希望同一只橘猫能形成稳定的视觉 IP。

**看到的现象：** 旧版入口说流程图和封面不加橘猫；流程图 reference 允许在部分节点旁加猫；封面模板又把橘猫设成固定主视觉。

如果只把规则统一成“所有图都加猫”，问题仍然没有解决。流程图里多一只站在角落的猫，只会遮挡信息。

**根本原因：** 规则按图片类型决定“加不加猫”，却没有判断橘猫在画面里承担什么功能。

**实际处理：** 新规则要求生成前先回答三个问题：橘猫代表谁？正在做什么？它怎样帮助理解主题？答不出来就不加。背景、角落贴纸、品牌水印和单纯卖萌都不算合理用途。

**验证结果：** 叙事图、流程图、封面、设计系统和示例都加入了同一项功能性检查。比如这篇的封面里，橘猫不是站在标题旁边，而是在把冲突规则整理成分类 reference，它的动作就是文章主题。

**怎么避免：** 视觉 IP 不是每张图都出现，而是每次出现都有事情可做。

以后生成任何正文图前，我会先过这三问：

1. 它代表谁：作者、读者、系统状态，还是流程里的某个角色？
2. 它正在做什么：触发、选择、等待、对比、接收结果，还是修复问题？
3. 它帮助理解什么：解释概念、显示状态、串起流程，还是承接情绪？

三问答不出来，说明这张图不该加猫，或者这张图本身没有信息价值。

## 最后验证：不要把工具失败误判成项目失败

> **注：** 以下验证步骤是本次 **Skill 代码库重构**的专属检查，针对的是 `.md` 文件一致性和 YAML 格式正确性。这不是公众号文章的通用发布前检查，文章发布前检查见 `SKILL.md` 的完稿自检清单。

我没有只看文件行数。最后实际执行了四类检查：

1. `git diff --check`：检查空白错误和补丁格式
2. YAML 解析：确认 frontmatter 只有 `name` 和 `description`
3. 相对链接扫描：确认 README 和 reference 链接都能找到文件
4. 旧规则关键词扫描：确认冲突颜色、旧标题选择流程和装饰性橘猫指令没有残留

这里还有一个小插曲：官方 `quick_validate.py` 在当前环境缺少 `PyYAML`，不能直接运行。我没有把“脚本启动失败”写成“Skill 校验失败”，而是用系统自带的 Ruby YAML 解析器完成等价的 frontmatter 检查，并在交付说明里保留这个限制。

**验证工具失败，和被验证对象失败，是两件不同的事。** 这也是这次最值得留下的实战细节。

如果你也遇到类似情况，可以按这个顺序判断：

| 现象 | 先判断 | 下一步 |
|------|--------|--------|
| 校验脚本跑不起来 | 是项目错，还是校验环境缺依赖 | 用等价轻量方法验证核心目标 |
| 链接扫描失败 | 是链接不存在，还是路径基准错了 | 固定从仓库根目录解析 |
| 关键词扫描命中 | 是旧规则残留，还是历史说明 | 保留必要历史，删除执行性旧规则 |
| 示例和规则不一致 | 是示例过期，还是规则过严 | 优先同步示例，因为模型会学示例 |

## 一份可以复用的检查清单

这次重构让我更确定一件事：**Skill 的质量不取决于写了多少规则，而取决于每条规则有没有唯一、清晰、可验证的执行路径。**

如果你正在维护一个越来越长的 Skill，可以按这份清单检查：

- 入口是否只保留决策和路由，而不是塞满所有细节？
- 同一条规则是否在入口、reference、README 和示例中保持一致？
- 能力不可用时，有没有明确的降级结果？
- 示例展示的行为，是否真的是规则要求的行为？
- 校验失败时，分清是工具环境问题还是 Skill 本身有问题了吗？
- 每个 reference 是否只负责一个主题，而不是顺手定义全局规则？
- 每条硬规则是否有对应的验证方法？

最后留一个具体问题：你现在的 Skill 里，最容易“一件事多个文件各说各话”的规则是哪一类？是输出格式、工具能力、示例，还是失败降级？
